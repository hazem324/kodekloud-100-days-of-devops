#   Jenkins Deployment Job

The Nautilus DevOps team implemented a **CI/CD pipeline using Jenkins and Gitea** to automatically deploy a web application whenever new changes are pushed to the repository.

This setup ensures:

*  Automatic deployment on every push
*  Continuous delivery
*  Proper use of Jenkins **agent (node)** architecture

---

##  Task Overview

* Repository hosted on **Gitea**
* Code cloned automatically by Jenkins
* Job executed on **App Server (agent node)**
* Deployment target:

```bash
/var/www/html
```

* Application served using **Apache (httpd)** on port **8080**

---

##  Architecture

```text
Developer → Gitea → Webhook → Jenkins → Agent (stapp01) → Apache → User
```

 Jenkins agent executes the job remotely and deploys the app.

---

| Username              | sarah           |

 Jenkins agents execute jobs on remote machines instead of the controller

#### Job Name:


####  Build Step → Execute Shell

```bash
#!/bin/bash

echo "Running on agent: $(hostname)"

# Go to Jenkins workspace
cd $WORKSPACE

# Clean old deployment
sudo rm -rf /var/www/html/*

# Copy all files to web directory
sudo cp -r * /var/www/html/

# Fix permissions
sudo chown -R sarah:sarah /var/www/html

# Restart Apache
sudo systemctl restart httpd

echo "Deployment completed successfully"
```

#  Good to Know 

---

##  Webhook-Triggered Deployments

* **Automatic Triggers**: Deployment starts automatically when code is pushed
* **Event-Driven Architecture**: Actions are triggered by events (e.g. Git push)
* **Real-Time Execution**: No delay — Jenkins reacts instantly to changes
* **Continuous Deployment**: Enables fully automated delivery pipeline

 A webhook is an **HTTP callback that sends data when an event occurs** ([Apideck][1])

---

##  How Webhooks Work (In This Task)

```text
Developer push → Gitea → Webhook → Jenkins → Job execution → Deployment
```

* Gitea detects a **push event**
* Sends HTTP request to Jenkins webhook URL
* Jenkins triggers the job automatically

 This avoids polling and makes the system efficient and reactive ([Red Hat][2])

---

##  Webhook Configuration

* **Payload URL**:
  Jenkins endpoint receiving events

  ```text
  http://<jenkins-url>:8080/github-webhook/
  ```

* **Content Type**: JSON (standard format)

* **Event Types**:

  * Push 
  * Pull Request
  * Tag creation

* **Branch Filtering**:

  * Trigger only on `master` branch
  * Avoid unnecessary builds

---

##  CI/CD Pipeline Integration

* **Source Control**: Gitea repository stores code
* **Trigger**: Webhook detects changes
* **Build**: Jenkins job runs automatically
* **Deploy**: Files copied to `/var/www/html`
* **Serve**: Apache (httpd) delivers application

 Jenkins pipelines can be triggered by webhooks to automate builds and deployments ([Wikipedia][3])

---

##  Why Use Webhooks?

*  Faster than polling (no repeated checks)
*  Instant response to code changes
*  Fully automated pipeline
*  No manual job execution needed

 Webhooks allow systems to notify others instantly when events occur 

---

##  Why Use Agent?

* Run jobs on target server
* Avoid SSH complexity
* Better CI/CD architecture
* Distributed builds

##  Security Considerations

* **API Tokens**: Secure communication between Gitea and Jenkins
* **Authentication**: Prevent unauthorized webhook calls
* **Secret Tokens**: Validate webhook source
* **Access Control**: Restrict Jenkins endpoints
* **Audit Logs**: Track deployment activity

---

##  Pro Tip 

* Always test webhook delivery from Gitea
* Check Jenkins logs if trigger fails
* Use `$WORKSPACE` for deployment (not manual repo path)
