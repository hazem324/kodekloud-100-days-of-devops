# Good to Know

## Project-Level Security

Jenkins allows **project-based security**, which means permissions can be configured **per job** instead of globally.

Key advantages:

* **Granular Control** – Manage permissions for each Jenkins job independently.
* **Inheritance** – Jobs can inherit global permissions using **Parent ACL**.
* **Override Capability** – Additional permissions can be granted for specific users at the job level.
* **Team Isolation** – Different teams can access only the jobs they are responsible for.

## Permission Matrix

The **Matrix Authorization Strategy** allows administrators to assign permissions to **users or groups** through a permission grid.

Important concepts:

* **Users and Groups** – Permissions can be assigned to individual users or groups.
* **Permission Categories** – Permissions are grouped into categories such as Job, Run, and SCM.
* **Inheritance Strategy** – Determines whether permissions inherit from global Jenkins security.
* **Fine-Grained Access** – Each permission can be controlled independently.

## Security Best Practices

When managing Jenkins permissions, follow these DevOps security practices:

* **Principle of Least Privilege** – Grant only the permissions required for the user's role.
* **Use Groups or Roles** – Managing groups is easier than managing individual users.
* **Avoid Global Admin Access** – Restrict admin permissions to trusted users only.
* **Periodic Permission Review** – Regularly audit access to Jenkins jobs and pipelines.

## Permission Categories

| Category | Description                                      |
| -------- | ------------------------------------------------ |
| Job      | Permissions related to Jenkins job configuration |
| Run      | Permissions related to build execution           |
| SCM      | Permissions related to source control operations |

## Common Jenkins Permissions

| Permission | Purpose                                  |
| ---------- | ---------------------------------------- |
| Read       | View job configuration and build history |
| Build      | Trigger builds                           |
| Configure  | Modify job settings                      |
| Cancel     | Stop running builds                      |
| Update     | Modify build run data                    |
| Tag        | Tag builds in SCM                        |

## DevOps Best Practices

* Apply the **Principle of Least Privilege**
* Avoid granting **global admin permissions**
* Use **role-based access control for teams**
* Regularly audit user permissions
