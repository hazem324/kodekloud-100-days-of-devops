# Jenkins Conditional Pipeline

This task involves creating a Jenkins pipeline that deploys a static website from a Gitea repository to the Nautilus App Server using a Jenkins agent.

The repository `web_app` already exists on **App Server 1** under:

```
/var/www/html
```

The pipeline must deploy either the **master** or **feature** branch based on a Jenkins parameter.

---

# Task Requirements

* Create a Jenkins agent named **App Server 1**
* Label the agent as **stapp01**
* Remote root directory:

```
/home/sarah/jenkins_agent
```

* The repository already exists on the server:

```
/var/www/html
```

* Apache is running on:

```
port 8080
```

* Create a Jenkins pipeline job named:

```
datacenter-webapp-job
```

The pipeline must:

* Accept a parameter **BRANCH**
* Deploy **master** if `master` is passed
* Deploy **feature** if `feature` is passed
* Use a single stage named **Deploy**


| Type             | Name   |
| ---------------- | ------ |
| String Parameter | BRANCH |

```

```

### 4.2.Pipeline Script

```groovy
pipeline {
    agent { label 'stapp01' }

    parameters {
        string(name: 'BRANCH', defaultValue: 'master', description: 'Enter branch name: master or feature')
    }

    stages {
        stage('Deploy') {
            steps {
                script {
                    if (params.BRANCH == 'master') {
                        sh '''
                        cd /var/www/html
                        git fetch --all
                        git checkout master
                        git pull origin master
                        '''
                    } 
                    else if (params.BRANCH == 'feature') {
                        sh '''
                        cd /var/www/html
                        git fetch --all
                        git checkout feature
                        git pull origin feature
                        '''
                    } 
                    else {
                        error "Invalid branch name. Use 'master' or 'feature'."
                    }
                }
            }
        }
    }
}
```

## Good to Know

### Jenkins Agents (Slave Nodes)

* **Distributed Builds**: Jenkins agents allow jobs to run on remote machines instead of the Jenkins master.
* **Labels**: Labels like `ststor01` help target specific agents when running jobs.
* **Remote Workspace**: The remote root directory (e.g., `/var/www/html`) acts as the working directory for builds executed on the agent.

### Parameterized Pipelines

* **Dynamic Deployment**: Parameters such as `BRANCH` allow selecting which code branch to deploy at runtime.
* **Flexible CI/CD**: One pipeline can support multiple deployment scenarios without creating multiple jobs.
* **User Input**: Developers can control pipeline behavior directly from the Jenkins UI.

### Conditional Deployment Logic

* **Branch-Based Deployment**: Different branches (e.g., `master`, `feature`) can trigger different deployment behaviors.
* **Runtime Decisions**: Pipelines evaluate parameter values during execution to decide which actions to perform.
* **Controlled Releases**: Conditional logic helps manage staging features or experimental code separately from production-ready code.

### Shared Storage Deployment

* **Centralized Code Location**: Storing the repository under `/var/www/html` on the storage server ensures consistency across multiple application servers.
* **Mounted Volumes**: Since the storage path is mounted to app servers, updates automatically reflect across all servers.
* **Simplified Deployment**: Deploying code once to shared storage eliminates the need to copy files to each app server individually.

### Static Web Deployment

* **Document Root**: Web servers like Apache serve files directly from the document root (`/var/www/html`).
* **No Subdirectory Deployment**: Applications must be deployed directly to the root to ensure the site loads from the main URL.
* **Immediate Updates**: Static files become available instantly after deployment without restarting the web server.
